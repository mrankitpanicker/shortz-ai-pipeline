Quick Start
Running in two minutes
Docker is the recommended path — it handles Redis, the GPU worker, and the API server in a single command.
→ One docker compose up starts all five services in the correct order with health gates between each.
1
Clone and start
bash
git clone https://github.com/youruser/shortz.git && cd shortz
docker compose up --build

Starts Redis, API server, and GPU worker in correct order with health gates.

2
Generate a video
bash
curl -X POST http://localhost:8000/generate \
  -H "Content-Type: application/json" \
  -d '{"count": 1, "voice_path": "voices/uvi.wav"}'
3
Poll job status
bash
curl http://localhost:8000/status/{job_id}
# → {"status":"running","stage":"voice","progress":42}
1
Prerequisites & install
bash — prereqs: Python 3.11+, Redis 6.2+, FFmpeg, CUDA toolkit
python -m venv venv && source venv/bin/activate
pip install -r requirements.txt
2
Start all services via supervisor
bash
python system/shortz_supervisor_v2.py
# Launches: Redis → Worker → API → GUI → Monitor
# With health gates and restart backoff
3
Or run services manually
bash — 4 terminals
redis-server                                # T1
python worker.py                             # T2: waits for XTTS load
python -m uvicorn api_server:app --port 8000  # T3
python services/gui_main.py                  # T4: production GUI
1
Start full observability stack
bash
docker compose \
  -f docker-compose.yml \
  -f docker/docker-compose.monitoring.yml \
  up -d

GPU Lifecycle

    
Model load / unload sequence

    
XTTS and Whisper cannot coexist on 4 GB VRAM. The pipeline manages VRAM as a resource — loading each model only when needed, releasing it immediately after use.

    
→ XTTS stays resident across jobs (saves 8 s/job). Whisper loads per-job and is freed immediately — total peak is ~3 GB, leaving 1 GB headroom.

Constraints

    
Honest trade-offs

    
Every architectural decision here was forced by hardware. These are the constraints, and the designs they drove.

    
→ The scale-out path is simple: add more workers across more GPUs. The Redis queue handles coordination automatically.

  


  

    

      ⚡
      
XTTS + Whisper cannot coexist

      
XTTS v2 uses ~2 GB and Whisper Small ~1 GB. Together they leave no headroom on 4 GB VRAM. The pipeline enforces sequential loading: XTTS stays resident across jobs; Whisper loads per-job and is immediately freed after alignment.

      Hardware limit
    


    

      ⏱
      
Sequential = no pipeline parallelism

      
Jobs are processed one at a time. A batch of 10 waits in the Redis queue — they do not run in parallel. This is intentional: GPU memory is the bottleneck, not CPU or I/O. Multiple workers across multiple GPUs is the scale-out path.

      Design trade-off
    


    

      🔁
      
XTTS startup latency is amortised

      
Loading XTTS takes ~8 seconds. Rather than loading it per-job (which would add 8 s to every video), the worker loads XTTS once at startup and keeps it resident. Only Whisper loads/unloads per-job (fast: ~2 s).

      Design decision
    


    

      📦
      
Single Redis as the coordination layer

      
All inter-process state — job status, progress, stage, voice path — lives in Redis hashes. No shared memory, no file locks. This makes it trivially horizontally scalable: add workers by running more Docker containers.

      Scalability
    


    

      🖥
      
Legacy GUI mode is dangerous

      
The legacy main.pyw path loads XTTS directly in the GUI process. On 4 GB VRAM, this conflicts with the worker and causes OOM crashes. Always use services/gui_main.py (API-bridged mode).

      Known hazard
    


    

      ⚙
      
FFmpeg GPU acceleration is optional

      
h264_nvenc is used when available, but FFmpeg falls back to libx264 on CPU automatically. This means the render stage works on CPU-only machines — useful for testing without a CUDA-capable GPU present.

      Graceful degradation

Queue operations

        

          
redis — critical ops

          
# Enqueue (atomic pipeline)
HSET job:{id} status queued progress 0 stage waiting created {ts}
RPUSH shortz_jobs {id}

# Dequeue (BLMOVE — Redis 6.2+, blocks until job available)
BLMOVE shortz_jobs shortz_processing RIGHT LEFT 0

# Fallback (Redis <6.2)
BRPOPLPUSH shortz_jobs shortz_processing 0

# Complete / cleanup
HSET job:{id} status complete progress 100 stage done
LREM shortz_processing 1 {id}

FastAPI — async model

        

          Runs on uvicorn's async event loop. Blocking Redis calls are offloaded via asyncio.to_thread() — the event loop never blocks. Redis client is lazy-initialised on first request, not at import — this lets the API start before Redis is ready.
        

        

        
Worker — synchronous blocking loop

        

          
pseudocode

          
while True:
    PING redis # verify connection before blocking
    job_id = BLMOVE(shortz_jobs, shortz_processing)  # blocks
    try:
        run_pipeline(job_id)   # Text → XTTS → Whisper → ASS → FFmpeg
        HSET job:{id} status=complete
    except:
        HSET job:{id} status=failed error=traceback
        reconnect_with_backoff()
    LREM shortz_processing job_id

API Layer

            
Global exception handler catches all unhandled errors → returns structured 500 JSON. API never crashes — uvicorn process stays alive.

          

          

            
Worker

            
Per-job try/except — failed jobs are marked in Redis (status=failed, error=traceback) and removed from shortz_processing. Redis reconnect with exponential backoff.

          

          

            
GUI

            
Per-thread error signals with exponential poll backoff. QueuedConnection for all signal-slot wiring ensures thread safety. QMutex protects all shared state.

          

          

            
Supervisor

            
Restart backoff: 2 s → 4 s → 8 s → 60 s max. Maximum 10 attempts per service before giving up and alerting operator. All restarts logged to system.log.

          

        


        

        
Reliability guarantee: BLMOVE + processing list

        

          Jobs are moved — not consumed — from shortz_jobs to shortz_processing atomically. If a worker crashes mid-job, the job ID remains in shortz_processing and can be recovered by an operator or a future requeue sweep. This is the Redis "reliable queue" pattern.
